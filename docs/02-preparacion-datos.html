2.1 Objetivos del capítulo

Al finalizar este capítulo, el lector será capaz de:

  • Explicar por qué la preparación de datos es una etapa fundamental en aprendizaje automático.
  • Distinguir entre datos crudos y datos preparados.
  • Leer un conjunto de datos real en R.
  • Revisar la estructura general de una base de datos.
  • Seleccionar variables relevantes para un problema de clasificación.
  • Crear una variable respuesta a partir de una variable original.
  • Guardar una base preparada para usarla en capítulos posteriores.

2.2 Introducción

En muchos cursos de estadística, programación o aprendizaje automático se utilizan bases de datos pequeñas, limpias y listas para trabajar. Esto es útil para aprender los conceptos básicos, pero en la práctica profesional la situación suele ser muy diferente.

Los datos reales generalmente vienen con problemas como:

  • nombres de variables poco claros,
  • códigos numéricos que representan categorías,
  • valores faltantes,
  • columnas que no se necesitan,
  • variables con tipos incorrectos,
  • archivos grandes,
  • fechas y horas separadas,
  • registros incompletos,
  • categorías escritas de distintas formas.

Por eso, antes de aplicar cualquier algoritmo de aprendizaje automático, es necesario preparar los datos.

A esta etapa se le conoce como preparación de datos o data preprocessing.

En términos sencillos:

Preparar datos significa transformar una base real, posiblemente desordenada, en una tabla limpia, clara y útil para construir modelos.

2.3 ¿Por qué es tan importante preparar los datos?

Un modelo de aprendizaje automático no puede corregir por sí solo todos los problemas de una base de datos.

Si los datos están mal preparados, el modelo puede producir resultados equivocados, aunque el algoritmo sea sofisticado.

Una frase común en ciencia de datos es:

Basura entra, basura sale.

Esto significa que si usamos datos mal organizados, incompletos o incorrectos, el resultado del modelo también será poco confiable.

Por ejemplo, imaginemos que queremos predecir si un accidente de tránsito tuvo víctimas, pero tenemos problemas como estos:

  • la hora del accidente aparece como texto en algunos registros y como número en otros;
  • algunos accidentes no tienen registrada la causa;
  • la variable de severidad está codificada con números que no entendemos;
  • hay columnas que no aportan información al modelo;
  • algunas variables representan categorías, pero R las interpreta como números.

Antes de modelar, debemos revisar y corregir este tipo de situaciones.

2.4 Datos crudos y datos preparados

Llamaremos datos crudos a la base tal como se obtiene de la fuente original.

Los datos crudos son valiosos porque contienen la información original, pero no siempre están listos para aplicar aprendizaje automático.

Llamaremos datos preparados a la base que ya fue revisada, transformada y organizada para un propósito específico.

La diferencia puede verse así:

Datos crudos
    ↓
Revisión
    ↓
Limpieza
    ↓
Selección de variables
    ↓
Transformación
    ↓
Datos preparados
    ↓
Modelo de aprendizaje automático

En este capítulo construiremos nuestra primera base preparada.

2.5 Conjunto de datos de trabajo: ATUS

Para iniciar usaremos un conjunto de datos real de México: Accidentes de Tránsito Terrestre en Zonas Urbanas y Suburbanas, conocido como ATUS, publicado por INEGI.

Este conjunto contiene información sobre accidentes de tránsito ocurridos en zonas urbanas y suburbanas.

Nos interesa porque permite construir problemas reales de clasificación, por ejemplo:

  • clasificar si un accidente tuvo víctimas;
  • clasificar el tipo de accidente;
  • identificar patrones por hora, día o mes;
  • estudiar diferencias entre municipios o entidades;
  • analizar causas frecuentes de accidentes.

En esta primera edición del libro usaremos ATUS como base para varios capítulos iniciales, porque permite explicar de forma didáctica el flujo completo de un proyecto de aprendizaje automático.

2.6 Pregunta de aprendizaje automático

Antes de tocar código, debemos plantear una pregunta.

En este capítulo trabajaremos con la siguiente pregunta:

¿Podemos preparar una base de datos que permita clasificar si un accidente de tránsito tuvo víctimas o fue solamente de daños materiales?

Esta pregunta nos lleva a definir dos elementos:

  1. La variable que queremos predecir.
  2. Las variables que usaremos para intentar predecirla.

2.7 Variable respuesta

La variable respuesta será:

accidente_con_victimas

Esta variable tendrá dos posibles valores:

Con víctimas
Solo daños

La construiremos a partir de la variable original que describe la clase del accidente.

En términos de aprendizaje automático, este será un problema de clasificación binaria, porque la respuesta solo tiene dos categorías.

2.8 Variables predictoras

Para iniciar usaremos pocas variables predictoras. La idea no es construir todavía el mejor modelo posible, sino preparar una base clara y manejable.

Usaremos variables como:

Variable Interpretación
MES Mes en que ocurrió el accidente
ID_HORA Hora del accidente
DIASEMANA Día de la semana
TIPACCID Tipo de accidente
CAUSAACCI Causa presunta del accidente

Estas variables son útiles porque permiten explorar preguntas como:

  • ¿hay más accidentes con víctimas en ciertas horas?
  • ¿el día de la semana influye?
  • ¿algunos tipos de accidente son más graves que otros?
  • ¿la causa presunta está relacionada con la severidad?

Para este primer ejemplo usaremos el siguiente diccionario didáctico:

2.20 Crear la variable respuesta

Ahora crearemos una nueva variable llamada:

accidente_con_victimas

La idea será clasificar los accidentes en dos grupos:

Con víctimas
Solo daños

Usaremos case_when() para construir la variable.

Code 
if (!is.null(atus) && "CLASACC" %in% names(atus_base)) {
  atus_modelo <- atus_base |>
    mutate(
      CLASACC_TXT = as.character(CLASACC),
      CLASACC_TXT = str_to_lower(CLASACC_TXT),
      accidente_con_victimas = case_when(
        str_detect(CLASACC_TXT, "sólo daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "solo daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daã") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daÃ") ~ "Solo daños",
        str_detect(CLASACC_TXT, "no fatal") ~ "Con víctimas",
        str_detect(CLASACC_TXT, "fatal") ~ "Con víctimas",
        CLASACC_TXT %in% c("1", "2") ~ "Con víctimas",
        CLASACC_TXT %in% c("3") ~ "Solo daños",
        TRUE ~ NA_character_
      )
    )
  table(atus_modelo$accidente_con_victimas, useNA = "ifany")
}

2.21 Comentario didáctico sobre la variable respuesta

Crear una variable respuesta es una decisión importante.

No basta con aplicar un algoritmo. Primero debemos definir claramente qué queremos predecir.

En este caso, convertimos la clase original del accidente en una variable binaria. Esto simplifica el problema y permite iniciar con modelos de clasificación sencillos.

Más adelante podríamos trabajar con una variable de tres categorías, por ejemplo:

Fatal
No fatal
Solo daños

Pero para una primera aproximación es conveniente empezar con dos clases.

2.23 Convertir variables categóricas

Algunas variables son códigos, pero deben tratarse como categorías.

Por ejemplo, ID_ENTIDAD es una clave de entidad federativa. Aunque tenga números, no debe interpretarse como una cantidad.

No tiene sentido decir que la entidad 20 es “el doble” de la entidad 10.

Por eso convertimos algunas variables a factor.

Code 
if (exists("atus_modelo")) {
  atus_modelo <- atus_modelo |>
    mutate(
      ID_ENTIDAD = as.factor(ID_ENTIDAD),
      ID_MUNICIPIO = as.factor(ID_MUNICIPIO),
      MES = as.factor(MES),
      DIASEMANA = as.factor(DIASEMANA),
      TIPACCID = as.factor(TIPACCID),
      CAUSAACCI = as.factor(CAUSAACCI),
      accidente_con_victimas = as.factor(accidente_con_victimas)
    )
  
  str(atus_modelo)
}

## Introducción

En muchos cursos de estadística, programación o aprendizaje automático se utilizan bases de datos pequeñas, limpias y listas para trabajar. Esto es útil para aprender los conceptos básicos, pero en la práctica profesional la situación suele ser muy diferente.

Los datos reales generalmente vienen con problemas como:

- nombres de variables poco claros,
- códigos numéricos que representan categorías,
- valores faltantes,
- columnas que no se necesitan,
- variables con tipos incorrectos,
- archivos grandes,
- fechas y horas separadas,
- registros incompletos,
- categorías escritas de distintas formas.

Por eso, antes de aplicar cualquier algoritmo de aprendizaje automático, es necesario preparar los datos.

A esta etapa se le conoce como **preparación de datos** o *data preprocessing*.

En términos sencillos:

> Preparar datos significa transformar una base real, posiblemente desordenada, en una tabla limpia, clara y útil para construir modelos.

## ¿Por qué es tan importante preparar los datos?

Un modelo de aprendizaje automático no puede corregir por sí solo todos los problemas de una base de datos.

Si los datos están mal preparados, el modelo puede producir resultados equivocados, aunque el algoritmo sea sofisticado.

Una frase común en ciencia de datos es:

```text
Basura entra, basura sale.
```

Esto significa que si usamos datos mal organizados, incompletos o incorrectos, el resultado del modelo también será poco confiable.

Por ejemplo, imaginemos que queremos predecir si un accidente de tránsito tuvo víctimas, pero tenemos problemas como estos:

- la hora del accidente aparece como texto en algunos registros y como número en otros;
- algunos accidentes no tienen registrada la causa;
- la variable de severidad está codificada con números que no entendemos;
- hay columnas que no aportan información al modelo;
- algunas variables representan categorías, pero R las interpreta como números.

Antes de modelar, debemos revisar y corregir este tipo de situaciones.

## Datos crudos y datos preparados

Llamaremos **datos crudos** a la base tal como se obtiene de la fuente original.

Los datos crudos son valiosos porque contienen la información original, pero no siempre están listos para aplicar aprendizaje automático.

Llamaremos **datos preparados** a la base que ya fue revisada, transformada y organizada para un propósito específico.

La diferencia puede verse así:

```text
Datos crudos
    ↓
Revisión
    ↓
Limpieza
    ↓
Selección de variables
    ↓
Transformación
    ↓
Datos preparados
    ↓
Modelo de aprendizaje automático
```

En este capítulo construiremos nuestra primera base preparada.

## Crear la variable respuesta

Ahora crearemos una nueva variable llamada:

```text
accidente_con_victimas
```

La idea será clasificar los accidentes en dos grupos:

```text
Con víctimas
Solo daños
```

Usaremos `case_when()` para construir la variable.

```{r}
if (!is.null(atus) && "CLASACC" %in% names(atus_base)) {
  atus_modelo <- atus_base |>
    mutate(
      CLASACC_TXT = as.character(CLASACC),
      CLASACC_TXT = str_to_lower(CLASACC_TXT),
      accidente_con_victimas = case_when(
        str_detect(CLASACC_TXT, "sólo daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "solo daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daños") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daã") ~ "Solo daños",
        str_detect(CLASACC_TXT, "daÃ") ~ "Solo daños",
        str_detect(CLASACC_TXT, "no fatal") ~ "Con víctimas",
        str_detect(CLASACC_TXT, "fatal") ~ "Con víctimas",
        CLASACC_TXT %in% c("1", "2") ~ "Con víctimas",
        CLASACC_TXT %in% c("3") ~ "Solo daños",
        TRUE ~ NA_character_
      )
    )
  table(atus_modelo$accidente_con_victimas, useNA = "ifany")
}
```

## Comentario didáctico sobre la variable respuesta

Crear una variable respuesta es una decisión importante.

No basta con aplicar un algoritmo. Primero debemos definir claramente qué queremos predecir.

En este caso, convertimos la clase original del accidente en una variable binaria. Esto simplifica el problema y permite iniciar con modelos de clasificación sencillos.

Más adelante podríamos trabajar con una variable de tres categorías, por ejemplo:

```text
Fatal
No fatal
Solo daños
```

Pero para una primera aproximación es conveniente empezar con dos clases.

## Eliminar registros sin respuesta

Para entrenar un modelo, necesitamos que cada registro tenga una clase conocida.

Por eso eliminamos registros donde no se pudo construir la variable respuesta.

```{r}
if (exists("atus_modelo")) {
  atus_modelo <- atus_modelo |>
    filter(!is.na(accidente_con_victimas))

}
```

## Convertir variables categóricas

Algunas variables son códigos, pero deben tratarse como categorías.

Por ejemplo, `ID_ENTIDAD` es una clave de entidad federativa. Aunque tenga números, no debe interpretarse como una cantidad.

No tiene sentido decir que la entidad 20 es “el doble” de la entidad 10.

Por eso convertimos algunas variables a factor.

```{r}
if (exists("atus_modelo")) {
  atus_modelo <- atus_modelo |>
    mutate(
      ID_ENTIDAD = as.factor(ID_ENTIDAD),
      ID_MUNICIPIO = as.factor(ID_MUNICIPIO),
      MES = as.factor(MES),
      DIASEMANA = as.factor(DIASEMANA),
      TIPACCID = as.factor(TIPACCID),
      CAUSAACCI = as.factor(CAUSAACCI),
      accidente_con_victimas = as.factor(accidente_con_victimas)
    )
  
  str(atus_modelo)
}
```
